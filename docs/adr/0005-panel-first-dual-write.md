# 0005 — Panel-first dual-write без 2PC

**Статус:** Accepted

## Контекст

Провижининг подписки затрагивает две системы без общей транзакции: локальную БД и панель Remnawave
(HTTP). Распределённой транзакции (2PC) нет. Нужен предсказуемый порядок и восстановление после сбоя.

## Решение

**Панель — первой, вне DB-транзакции**, затем локальный коммит. Порядок операций внутри:
**grant → mark-issued → notify**. Падение после удачного вызова панели, но до/во время локального
коммита оставляет «remote orphan» (есть на панели, нет локально) — это документированное окно.

Подстраховки:
- неудавшиеся panel-write после локального коммита → **panel-write retry queue** (taskiq),
  идемпотентно передрайвит `sync_subscription`;
- **reconcile-джоб** (`SyncFromRemnawave` / `SyncFromLocal`) чинит дрейф в обе стороны;
- операции панели идемпотентны по `short_id`↔`uuid`.

## Последствия

- (+) Пользователь не платит «в пустоту»: если панель упала — покупка не завершается.
- (+) Дрейф самозалечивается ретраями/reconcile.
- (−) Есть окно «remote orphan» — обязателен reconcile; синк должен быть идемпотентным.
- Никогда не оборачивать вызов панели в DB-транзакцию (держал бы соединение и всё равно не атомарно).
