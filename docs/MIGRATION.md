# Миграция с другого бота / панели

Кабинет → **Обслуживание → Миграция с другого бота**. Выбираешь источник, загружаешь файл
(или DSN), жмёшь «Проверить» — видишь счётчики, потом «Начать миграцию». Повторный запуск
безопасен: юзеры матчатся по `telegram_id`, подписки по uuid панели, платежи по внешнему id —
обновление, не дубли. Баланс уже существующего юзера повторный импорт не трогает.

## remnawave-shopbot

Файл `users.db` из папки бота. Переезжают балансы (оба кошелька суммируются), подписки с
ключами панели, платежи, промокоды, рефералы. Панель не трогаем — ключи юзеров продолжают
работать в момент переезда.

## Bedolaga

Любой из вариантов:

- `bot.db` — если бот работал в SQLite-режиме (`./data/bot.db`);
- `database.sql` из бэкап-архива, или **сам архив `backup_YYYYMMDD_HHMMSS.tar.gz`** —
  бот шлёт их в Telegram, либо лежат в `/app/data/backups`;
- `database.json` (ORM-дамп, если pg_dump не было);
- **Postgres DSN** из `.env` бота (`postgresql://remnawave_user:пароль@хост:5432/remnawave_bot`) —
  порт должен быть доступен с сервера HUB-BOT.

Переезжают: балансы (копейки 1:1), подписки с uuid панели (single- и multi-tariff режимы),
squad'ы, автоплатёж, платежи (включая маппинг шлюзов, pal24 → paypalych), промокоды
(баланс/дни/скидка-процент), рефералы. Панель не трогаем.

Не переезжает: promo-группы (промокоды типа `promo_group` пропускаются с пометкой),
кабинетные юзеры без `telegram_id`.

## RemnaShop (snoups/remnashop)

`db_backup_*.sql` из папки `backups` рядом с ботом (bind-mount `./backups`), либо DSN
(`postgresql://remnashop:пароль@127.0.0.1:5001/remnashop` — порт паблишится на localhost
сервера RemnaShop).

Переезжают: юзеры (скидки personal/purchase, триал-флаг), текущая подписка каждого юзера
с uuid панели и plan_snapshot, завершённые платежи (суммы из pricing с учётом валюты —
рубли ×100, звёзды 1:1), промокоды, рефералы. Панель не трогаем.

Не переезжает: кошелька в RemnaShop нет (points не конвертируются), web-юзеры без Telegram,
исторические подписки (только текущая — панель-юзер один на юзера).

## 3x-ui

Файл `/etc/x-ui/x-ui.db` (останови панель или сделай копию файла). Единственный источник,
где юзеров на панели Remnawave ещё нет — импорт **создаёт их**:

- `vlessUuid` = id клиента из 3x-ui → конфиги на устройствах продолжают работать
  (нужен тот же Reality/инбаунд-профиль на нодах, это настройка панели, не импорта);
- `shortUuid` = subId → старую ссылку подписки можно завести 301-редиректом
  `/sub/{subId} → https://SUB_DOMAIN/{subId}` (идентичность, т.к. shortUuid=subId);
- `tgId` клиента → юзер бота (официальный remnawave/migrate это теряет);
- `totalGB` (на деле байты) → лимит трафика, отрицательный `expiryTime`
  («после первого запуска») → срок = сейчас + остаток.

После «Проверить» выбери **сквад** — в него попадут созданные юзеры. Клиенты без `tgId`
создаются на панели и в базе без Telegram-привязки (видны в админке, счётчик «без Telegram»).

Триал импортированным из 3x-ui выключен (`is_trial_available = false`).

## Ограничения

- Файл до 200 МБ, хранится вне публичной статики, удаляется после успешного импорта.
- Импорт идёт в одном запросе: на десятках тысяч юзеров 3x-ui (создание на панели) может
  занять минуты — не закрывай вкладку.
- Все запуски пишутся в журнал операций (audit).
